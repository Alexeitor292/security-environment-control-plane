**Status of this document:** point-in-time implementation snapshot (descriptive).
**Governing document:** [`docs/PROJECT_CHARTER.md`](PROJECT_CHARTER.md).
**Authoring convergence (ADR-016) ‚Äî control-plane complete, simulated:** publishing an approved topology revision into a canonical immutable `EnvironmentVersion`, and planning against exactly that version, is now implemented end to end (PR A‚ÄìE) as fail-closed, org-scoped control-plane logic on a **simulated** planning path ‚Äî see [`ADR-016`](adr/ADR-016-authoring-convergence-environment-version-publication.md). Approval of a topology revision **publishes nothing automatically**; publication is one explicit, `version:publish`-gated, audited, idempotent mutation; a deployment plan binds **exactly one** immutable `EnvironmentVersion` (`environment_version_id` + `version_content_hash`) and surfaces its typed, server-owned publication provenance identically across generate ‚Üí submit ‚Üí approve, failing closed (redacted `plan_version_binding_invalid`) on any binding disagreement; legacy `v1alpha1` versions remain fully plannable with **null** provenance. Every step is a separate, explicit action ‚Äî no publication auto-creates an exercise, no exercise auto-generates a plan, no transition auto-triggers the next, and approval deploys nothing. PR E added **no** worker/provider/transport/real-infrastructure code and **no** DeploymentPlan topology/provenance column or migration; real mutation remains sealed. The point-in-time PR-A‚Ä¶PR-D lines below are retained as historical progression.

**Backend OIDC bearer verification (ADR-017 / OIDC-A) ‚Äî implemented:** the control-plane API now performs cryptographic OIDC access-token verification. A presented `Authorization: Bearer` token is accepted only when it is syntactically valid, uses the fixed `RS256` allowlist, its signature verifies against the configured issuer's JWKS, its `iss` exactly matches configuration, the configured API `aud` is present, it is unexpired with a valid `nbf`/`iat` (bounded clock skew), and its exact `sub` maps to exactly one **pre-provisioned** internal user (`app_user.subject`, now globally unique for non-null values via a partial index). Organization, roles, and permissions come **exclusively** from SECP database records ‚Äî the token's roles, groups, email, or organization claims grant **nothing**, and there is **no** just-in-time user creation. A Bearer header is always evaluated before the development fallback, and an invalid token never falls back to dev-admin; the dev fallback remains available only on a no-header request in non-production. Authentication failures are closed and redacted (`401 {"error":{"code":"unauthenticated"}}` + `WWW-Authenticate: Bearer`; a distinct `503 {"error":{"code":"authentication_unavailable"}}` when discovery/JWKS is unavailable), never revealing the cause; logs carry only a bounded reason category, never the token/claims/subject. The interactive browser login that obtains those tokens is implemented by **OIDC-B** (below). This slice establishes the trusted backend boundary only; it unseals **no** provisioning or infrastructure mutation, and authentication remains distinct from the still-authoritative per-route authorization checks.

**Browser Authorization Code + PKCE login (ADR-018 / OIDC-B) ‚Äî implemented:** the placeholder browser login is replaced by a real, provider-neutral OpenID Connect **Authorization Code flow with PKCE (S256)** through the public `secp-web` client (no client secret), using the maintained `oidc-client-ts` library. The browser reads a **public, secret-free** `GET /api/v1/auth/config` (server-derived `mode`, issuer, public client id, audience, fixed `openid profile email` scope with **no** `offline_access`, and fixed relative callback paths), runs the code+PKCE flow, and sends the resulting **access token** as `Authorization: Bearer` to the API ‚Äî the ID token is never an API credential and never grants org/role/permission. `/api/v1/me` is the **authoritative** browser identity; token claims are never used for SECP authorization or organization selection. Tokens are **session-scoped only** (no localStorage/IndexedDB/cookies/DB persistence, no service-worker cache); access-token expiry clears local state and requires a fresh interactive login (no silent renewal, no `offline_access`, no refresh-token session). Every application route is guarded (no protected content or domain API call before authentication resolves); an invalid callback, `state`/`nonce` mismatch, token error, or API **401** fails closed to `/login` with a sanitized same-origin return path, a **403** keeps the session, and a **503** shows a provider-unavailable message. Logout clears local state and invokes the provider end-session when available. Frontend auth errors are bounded categories that never expose a token, code, state, nonce, verifier, claim, subject, or provider detail. The development fallback remains available **only** when the backend reports `mode = dev_fallback` (non-production). This slice adds **no** backend session/BFF, token database, JIT provisioning, token-derived RBAC, or multi-issuer support, and unseals **no** provisioning or infrastructure.

**Production OIDC deployment guardrails, preflight & runbooks (ADR-019 / OIDC-C) ‚Äî implemented:** the authentication architecture is now safely deployable and operationally documented. A **same-origin** production model is locked (one canonical HTTPS public origin serves the web app and the `/api/` SECP API; browser callback/logout URLs derive from it): `SECP_PUBLIC_ORIGIN` is validated as an exact HTTPS origin, production **CORS is disabled** (empty allow-origins; the API uses stateless bearer tokens, never cookies) while development allows only the exact configured origin without credentials or wildcards, and a production **Host allowlist** (canonical host + optional internal health host, never `*`) fails closed with no `/health` bypass. A token-free operator **preflight** (`python -m secp_api.oidc_preflight`) reuses the OIDC-A hardened HTTP seam to check discovery/JWKS, exact issuer agreement, HTTPS endpoints, and a usable RSA signing key ‚Äî without logging in, obtaining a token, or touching the database/audit log ‚Äî with stable exit codes. The frontend API base is **same-origin** in a browser build (never a localhost fallback in production). Placeholder-only production reference guardrails (`infra/production/`) and a full operations runbook (`docs/runbooks/oidc-production.md`) are included. This **does not** certify the whole platform as production-ready: pre-provisioned internal identities remain required, real provider mutation and the complete real disposable-lab lifecycle remain sealed or incomplete, and approval never becomes execution because authentication is deployable. There is **no** first-class production identity-lifecycle API/UI/CLI in this slice ‚Äî a direct DBA `subject` binding is outside the SECP application mutation path and is **not** SECP-audited (creates no `AuditEvent`); required `sub` bindings must exist and be independently verified before rollout, and a transactional, authorization-gated, SECP-audited identity-administration workflow remains future work.
**Publication contract (PR A):** the `controlplane.security/v1alpha2` publication composition, canonical-topology reconstruction, whole-definition environment hash, and server-derived publication fingerprint are now **contract-complete** as a pure, database-free contract. Publication **persistence, API route, permission, audit, and UI are not implemented**; approval still does not publish; and this contract enables **no** plan generation, EnvironmentVersion creation, or infrastructure execution. `v1alpha1` is unchanged.

**Publication persistence & service (PR B):** the pure PR-A contract is now persisted by a transactional, org-scoped, fail-closed, idempotent control-plane service (`publish_version`) that composes an approved topology revision + its passing validation result into a new immutable `controlplane.security/v1alpha2` `EnvironmentVersion` under a `SELECT FOR UPDATE` template lock, guarded by a distinct `version:publish` permission and the new publication-binding columns/constraints/immutability trigger. Direct `create_version` persists only `v1alpha1`, so a caller-fabricated `v1alpha2` publication envelope cannot bypass the service, which creates **no** exercise, deployment plan, or workflow dispatch and contacts **no** infrastructure. `v1alpha1` is unchanged.
**Publication API & provenance read model (PR C):** the explicit publication mutation is now reachable through the control-plane API ‚Äî `POST /api/v1/environment-versions/publish` ‚Äî which requires `version:publish` and consumes only an **already-approved** topology revision + its passing validation + a non-topology `v1alpha2` definition, returning one immutable `EnvironmentVersion`. Creation returns **201** with a single atomic `version.published` audit; an exact retry is **idempotent** (200, same version id, no new row/number, no duplicate audit). Every service refusal is durably audited (`version.publish_refused`) and mapped to a closed per-code HTTP status; malformed requests are redacted to a generic code; no backend/exception/rejected-value text is ever returned. Typed publication provenance (mirrored, immutable columns incl. the server-owned fingerprint) is now visible in the `EnvironmentVersion` read model (null for legacy `v1alpha1`). Publication still creates **no** exercise, plan, or workflow and contacts **no** worker/provider/infrastructure; **approval still publishes nothing automatically**; there is **no** frontend publication workflow (PR D) and no planning-provenance/end-to-end wiring (PR E). Production OIDC bearer-token verification remains **absent** and is still a serious production-readiness blocker (any `Authorization` header is refused; only the dev fallback authenticates). The end-to-end authoring‚Üípublication‚Üíplan workflow remains **incomplete**.
**Publication workflow UI (PR D):** the publication mutation now has an explicit React workflow, entered **contextually** from an approved topology document (a "Publish version‚Ä¶" action appears in the durable topology panel only for the exact approved/current/passing/clean/non-historical posture with `version:publish`; there is **no** global nav item and approval never navigates or publishes automatically). The page (`/environment-publication/:documentId` ‚Äî only the document id is authoritative from the URL) re-resolves the exact approved revision, its passing validation, and the destination template/base from the **server** (never cached workspace or URL pins), authors a non-topology `v1alpha2` definition (source-derived drafts deep-clone the source and disclose that server-owned `spec.topology`/`spec.publicationProvenance` were excluded; caller-supplied forbidden keys are refused locally without stripping the user's text), gates review on fresh schema validation, and requires an explicit acknowledgement before publishing. Results distinguish **201 creation** ("new immutable version") from **200 idempotent replay** ("already existed" ‚Äî never claimed as new). Typed publication provenance is shown in the Environment Library (server-owned `Version.publication_provenance`, null for legacy `v1alpha1`); "Create exercise" remains a separate action. The UI performs **no** exercise/plan/workflow creation and contacts **no** worker/provider/infrastructure; UI gating is usability only ‚Äî the publication service remains the authoritative permission and precondition boundary. Production OIDC remains **absent**; planning provenance and full end-to-end publication‚Üíplanning regression remain **future work** (PR E). The overall authoring-convergence workflow is **not** complete.

**Planning provenance & full end-to-end convergence (PR E):** the final PR wires publication provenance through planning and closes the authoring‚Üípublication‚Üíplanning loop, all on the **simulated** path. An exact, org-scoped, **read-only** `GET /api/v1/environment-versions/{version_id}` returns one immutable version (typed provenance for published `v1alpha2`, null for legacy `v1alpha1`) with no mutation, audit, topology-authoring lookup, or list-all/latest fallback ‚Äî the frontend resolves a source-derived publication base through it instead of scanning. `PlanOut` now carries a typed `environment_version_binding` (version number/id, template id, api version, content hash, and the same server-owned `publication_provenance`) derived from the exact bound version ‚Äî never from `plan.summary` or the spec ‚Äî and surfaced identically across generate/submit/approve. Every plan lifecycle mutation and every plan response first **re-verifies** the one-version binding (org scope + id + content-hash agreement + exercise cross-checks + a content-hash recompute) and **fails closed** with a redacted `plan_version_binding_invalid` (HTTP 409, no ids/hashes/spec leaked) on any disagreement. The `plan.generated` audit records safe version-binding lineage (`version_origin` = `published` | `legacy_manual`, plus published provenance ids/hashes) as **visibility metadata only** ‚Äî never a second canonical binding and never an authorization input. The plan-review UI shows an "Environment version binding" section (published-from-topology vs legacy/manual, typed provenance via hash chips, Environment Library deep-link) reading **only** the binding's provenance. The `DeploymentPlan` gained **no** topology/provenance column and there is **no** new migration; planning imports/queries **no** topology-authoring records and consumes only the immutable `version.spec`; approval creates **no** workflow/manifest and contacts **no** worker/provider/infrastructure. ## A. Status taxonomy

A capability is labelled with exactly one of the following (a capability may carry a
second qualifier such as `production-blocked` where a production validator or code seal
also applies):

| Label | Meaning |
| --- | --- |
| **implemented-simulated** | Fully implemented and exercised end-to-end, but every effect is a simulated record in the control-plane database via the simulator plugin. No real infrastructure is created. |
| **implemented-fake-only** | Implemented and exercised, but only against in-process fakes / injected test doubles. The real adapter is not the shipped runtime path and no real endpoint is contacted. |
| **controlled-live-read-only** | A real, **read-only** integration wired into the shipped runtime that can contact an explicitly authorized target ‚Äî but only after a complete, fail-closed gate chain passes. Sealed by default; never mutates. |
| **sealed-by-default** | The real implementation (or its architecture) exists in the tree but ships disabled and fail-closed: the shipped default composition refuses at the first privileged boundary. Reachable, if at all, only through an explicit, reviewed, deployment-local profile with every gate satisfied. |
| **contract-complete** | The durable schema, contracts, and code for a capability exist and are tested, but the capability is not wired into any shipped runtime path ‚Äî it is dormant. |
| **partially-implemented** | Part of the capability exists; the remainder does not. |
| **not-implemented** | No implementing code exists. The capability appears, if at all, only as names/strings in schemas, seed data, or test fixtures. |
| **production-blocked** | Cannot be activated in production by design: a production-config validator (`Settings`) or a hard code seal refuses it. |

## D. Critical distinctions

These separations are real and enforced in code; conflating them misstates current truth.

- **Local topology canvas state is not a saved revision.** Client edits are not persisted until a save writes an immutable `TopologyRevision`.
- **A saved revision is not necessarily validated.** A saved revision is `draft`; validation is a separate, permissioned action.
- **Validation is not submission.** Submission is refused unless the revision is validated with a current matching result.
- **Submission is not approval.** Approve/reject is a separate, separately-permissioned, audited decision on a submitted revision.
- **Approval is a decision only.** Approving a topology revision records a decision and sets the approved-revision pointer ‚Äî **nothing else**.
- **An approved topology revision is not yet a canonical published EnvironmentVersion.** Approving a topology revision does not publish a canonical EnvironmentVersion and does not generate a deployment plan.
- **EnvironmentVersion publication is not plan generation.** The plan generator binds to an `EnvironmentVersion`, not a topology revision.
- **Plan approval is not execution.** Approving a plan does not activate real provisioning (charter Invariant 5).
- **Controlled live read-only discovery is not provisioning, and is not the complete disposable-lab lifecycle.** It is read-only and imports no mutation code; the full real dry-run / apply / verify / destroy lifecycle does not exist yet.
- **Candidate plans produced by discovery are non-executable.** Live apply remains sealed.
- **The complete real dry-run / apply / verify / destroy lifecycle is not yet available.** It is sealed by default.
- **The B1-B disposable-lab architecture is locked (ADR-020) as a design-only architecture lock.** It defines the ordered real read-only qualification ‚Üí real OpenTofu plan ‚Üí exact change-set approval ‚Üí exact prepared-plan apply ‚Üí observed-state + isolation verification ‚Üí separate destroy plan ‚Üí separate destroy approval ‚Üí destroy ‚Üí zero-residue lifecycle, the operation-specific phased unsealing, the threat model, and the acceptance contract. **The architecture lock does not activate or authorize anything.** No real OpenTofu process has run, no real plan has run, no real apply or destroy has occurred, and no real Proxmox mutation has occurred. plan.
