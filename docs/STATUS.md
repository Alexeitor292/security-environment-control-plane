# Project Status — Current Capability Ledger

**Status of this document:** point-in-time implementation snapshot (descriptive).
**Governing document:** [`docs/PROJECT_CHARTER.md`](PROJECT_CHARTER.md).
**Authoring convergence (ADR-016) — control-plane complete, simulated:** publishing an approved topology revision into a canonical immutable `EnvironmentVersion`, and planning against exactly that version, is now implemented end to end (PR A–E) as fail-closed, org-scoped control-plane logic on a **simulated** planning path — see [`ADR-016`](adr/ADR-016-authoring-convergence-environment-version-publication.md). Approval of a topology revision **publishes nothing automatically**; publication is one explicit, `version:publish`-gated, audited, idempotent mutation; a deployment plan binds **exactly one** immutable `EnvironmentVersion` (`environment_version_id` + `version_content_hash`) and surfaces its typed, server-owned publication provenance identically across generate → submit → approve, failing closed (redacted `plan_version_binding_invalid`) on any binding disagreement; legacy `v1alpha1` versions remain fully plannable with **null** provenance. Every step is a separate, explicit action — no publication auto-creates an exercise, no exercise auto-generates a plan, no transition auto-triggers the next, and approval deploys nothing. PR E added **no** worker/provider/transport/real-infrastructure code and **no** DeploymentPlan topology/provenance column or migration; real mutation remains sealed. The point-in-time PR-A…PR-D lines below are retained as historical progression.

**Backend OIDC bearer verification (ADR-017 / OIDC-A) — implemented:** the control-plane API now performs cryptographic OIDC access-token verification. A presented `Authorization: Bearer` token is accepted only when it is syntactically valid, uses the fixed `RS256` allowlist, its signature verifies against the configured issuer's JWKS, its `iss` exactly matches configuration, the configured API `aud` is present, it is unexpired with a valid `nbf`/`iat` (bounded clock skew), and its exact `sub` maps to exactly one **pre-provisioned** internal user (`app_user.subject`, now globally unique for non-null values via a partial index). Organization, roles, and permissions come **exclusively** from SECP database records — the token's roles, groups, email, or organization claims grant **nothing**, and there is **no** just-in-time user creation. A Bearer header is always evaluated before the development fallback, and an invalid token never falls back to dev-admin; the dev fallback remains available only on a no-header request in non-production. Authentication failures are closed and redacted (`401 {"error":{"code":"unauthenticated"}}` + `WWW-Authenticate: Bearer`; a distinct `503 {"error":{"code":"authentication_unavailable"}}` when discovery/JWKS is unavailable), never revealing the cause; logs carry only a bounded reason category, never the token/claims/subject. The interactive browser login that obtains those tokens is implemented by **OIDC-B** (below). This slice establishes the trusted backend boundary only; it unseals **no** provisioning or infrastructure mutation, and authentication remains distinct from the still-authoritative per-route authorization checks.

**Browser Authorization Code + PKCE login (ADR-018 / OIDC-B) — implemented:** the placeholder browser login is replaced by a real, provider-neutral OpenID Connect **Authorization Code flow with PKCE (S256)** through the public `secp-web` client (no client secret), using the maintained `oidc-client-ts` library. The browser reads a **public, secret-free** `GET /api/v1/auth/config` (server-derived `mode`, issuer, public client id, audience, fixed `openid profile email` scope with **no** `offline_access`, and fixed relative callback paths), runs the code+PKCE flow, and sends the resulting **access token** as `Authorization: Bearer` to the API — the ID token is never an API credential and never grants org/role/permission. `/api/v1/me` is the **authoritative** browser identity; token claims are never used for SECP authorization or organization selection. Tokens are **session-scoped only** (no localStorage/IndexedDB/cookies/DB persistence, no service-worker cache); access-token expiry clears local state and requires a fresh interactive login (no silent renewal, no `offline_access`, no refresh-token session). Every application route is guarded (no protected content or domain API call before authentication resolves); an invalid callback, `state`/`nonce` mismatch, token error, or API **401** fails closed to `/login` with a sanitized same-origin return path, a **403** keeps the session, and a **503** shows a provider-unavailable message. Logout clears local state and invokes the provider end-session when available. Frontend auth errors are bounded categories that never expose a token, code, state, nonce, verifier, claim, subject, or provider detail. The development fallback remains available **only** when the backend reports `mode = dev_fallback` (non-production). This slice adds **no** backend session/BFF, token database, JIT provisioning, token-derived RBAC, or multi-issuer support, and unseals **no** provisioning or infrastructure.

**Production OIDC deployment guardrails, preflight & runbooks (ADR-019 / OIDC-C) — implemented:** the authentication architecture is now safely deployable and operationally documented. A **same-origin** production model is locked (one canonical HTTPS public origin serves the web app and the `/api/` SECP API; browser callback/logout URLs derive from it): `SECP_PUBLIC_ORIGIN` is validated as an exact HTTPS origin, production **CORS is disabled** (empty allow-origins; the API uses stateless bearer tokens, never cookies) while development allows only the exact configured origin without credentials or wildcards, and a production **Host allowlist** (canonical host + optional internal health host, never `*`) fails closed with no `/health` bypass. A token-free operator **preflight** (`python -m secp_api.oidc_preflight`) reuses the OIDC-A hardened HTTP seam to check discovery/JWKS, exact issuer agreement, HTTPS endpoints, and a usable RSA signing key — without logging in, obtaining a token, or touching the database/audit log — with stable exit codes. The frontend API base is **same-origin** in a browser build (never a localhost fallback in production). Placeholder-only production reference guardrails (`infra/production/`) and a full operations runbook (`docs/runbooks/oidc-production.md`) are included. This **does not** certify the whole platform as production-ready: pre-provisioned internal identities remain required, real provider mutation and the complete real disposable-lab lifecycle remain sealed or incomplete, and approval never becomes execution because authentication is deployable. There is **no** first-class production identity-lifecycle API/UI/CLI in this slice — a direct DBA `subject` binding is outside the SECP application mutation path and is **not** SECP-audited (creates no `AuditEvent`); required `sub` bindings must exist and be independently verified before rollout, and a transactional, authorization-gated, SECP-audited identity-administration workflow remains future work.
**Publication contract (PR A):** the `controlplane.security/v1alpha2` publication composition, canonical-topology reconstruction, whole-definition environment hash, and server-derived publication fingerprint are now **contract-complete** as a pure, database-free contract. Publication **persistence, API route, permission, audit, and UI are not implemented**; approval still does not publish; and this contract enables **no** plan generation, EnvironmentVersion creation, or infrastructure execution. `v1alpha1` is unchanged.

**Publication persistence & service (PR B):** the pure PR-A contract is now persisted by a transactional, org-scoped, fail-closed, idempotent control-plane service (`publish_version`) that composes an approved topology revision + its passing validation result into a new immutable `controlplane.security/v1alpha2` `EnvironmentVersion` under a `SELECT FOR UPDATE` template lock, guarded by a distinct `version:publish` permission and the new publication-binding columns/constraints/immutability trigger. Direct `create_version` persists only `v1alpha1`, so a caller-fabricated `v1alpha2` publication envelope cannot bypass the service, which creates **no** exercise, deployment plan, or workflow dispatch and contacts **no** infrastructure. `v1alpha1` is unchanged.
**Publication API & provenance read model (PR C):** the explicit publication mutation is now reachable through the control-plane API — `POST /api/v1/environment-versions/publish` — which requires `version:publish` and consumes only an **already-approved** topology revision + its passing validation + a non-topology `v1alpha2` definition, returning one immutable `EnvironmentVersion`. Creation returns **201** with a single atomic `version.published` audit; an exact retry is **idempotent** (200, same version id, no new row/number, no duplicate audit). Every service refusal is durably audited (`version.publish_refused`) and mapped to a closed per-code HTTP status; malformed requests are redacted to a generic code; no backend/exception/rejected-value text is ever returned. Typed publication provenance (mirrored, immutable columns incl. the server-owned fingerprint) is now visible in the `EnvironmentVersion` read model (null for legacy `v1alpha1`). Publication still creates **no** exercise, plan, or workflow and contacts **no** worker/provider/infrastructure; **approval still publishes nothing automatically**; there is **no** frontend publication workflow (PR D) and no planning-provenance/end-to-end wiring (PR E). Production OIDC bearer-token verification remains **absent** and is still a serious production-readiness blocker (any `Authorization` header is refused; only the dev fallback authenticates). The end-to-end authoring→publication→plan workflow remains **incomplete**.
**Publication workflow UI (PR D):** the publication mutation now has an explicit React workflow, entered **contextually** from an approved topology document (a "Publish version…" action appears in the durable topology panel only for the exact approved/current/passing/clean/non-historical posture with `version:publish`; there is **no** global nav item and approval never navigates or publishes automatically). The page (`/environment-publication/:documentId` — only the document id is authoritative from the URL) re-resolves the exact approved revision, its passing validation, and the destination template/base from the **server** (never cached workspace or URL pins), authors a non-topology `v1alpha2` definition (source-derived drafts deep-clone the source and disclose that server-owned `spec.topology`/`spec.publicationProvenance` were excluded; caller-supplied forbidden keys are refused locally without stripping the user's text), gates review on fresh schema validation, and requires an explicit acknowledgement before publishing. Results distinguish **201 creation** ("new immutable version") from **200 idempotent replay** ("already existed" — never claimed as new). Typed publication provenance is shown in the Environment Library (server-owned `Version.publication_provenance`, null for legacy `v1alpha1`); "Create exercise" remains a separate action. The UI performs **no** exercise/plan/workflow creation and contacts **no** worker/provider/infrastructure; UI gating is usability only — the publication service remains the authoritative permission and precondition boundary. Production OIDC remains **absent**; planning provenance and full end-to-end publication→planning regression remain **future work** (PR E). The overall authoring-convergence workflow is **not** complete.

**Planning provenance & full end-to-end convergence (PR E):** the final PR wires publication provenance through planning and closes the authoring→publication→planning loop, all on the **simulated** path. An exact, org-scoped, **read-only** `GET /api/v1/environment-versions/{version_id}` returns one immutable version (typed provenance for published `v1alpha2`, null for legacy `v1alpha1`) with no mutation, audit, topology-authoring lookup, or list-all/latest fallback — the frontend resolves a source-derived publication base through it instead of scanning. `PlanOut` now carries a typed `environment_version_binding` (version number/id, template id, api version, content hash, and the same server-owned `publication_provenance`) derived from the exact bound version — never from `plan.summary` or the spec — and surfaced identically across generate/submit/approve. Every plan lifecycle mutation and every plan response first **re-verifies** the one-version binding (org scope + id + content-hash agreement + exercise cross-checks + a content-hash recompute) and **fails closed** with a redacted `plan_version_binding_invalid` (HTTP 409, no ids/hashes/spec leaked) on any disagreement. The `plan.generated` audit records safe version-binding lineage (`version_origin` = `published` | `legacy_manual`, plus published provenance ids/hashes) as **visibility metadata only** — never a second canonical binding and never an authorization input. The plan-review UI shows an "Environment version binding" section (published-from-topology vs legacy/manual, typed provenance via hash chips, Environment Library deep-link) reading **only** the binding's provenance. A dedicated end-to-end regression drives template → topology draft → validate → submit → approve → publish → replay → create exercise → generate plan → submit → approve and asserts every step is explicit (no auto-exercise/plan/submit/approve/dispatch), publication is idempotent and side-effect-free, the plan binds exactly the published version with matching provenance, and legacy `v1alpha1` stays plannable with null provenance. The `DeploymentPlan` gained **no** topology/provenance column and there is **no** new migration; planning imports/queries **no** topology-authoring records and consumes only the immutable `version.spec`; approval creates **no** workflow/manifest and contacts **no** worker/provider/infrastructure. Production OIDC remains **absent**; real infrastructure remains sealed. This completes ADR-016 PR A–E on the control-plane/simulated path.

## A. Status taxonomy

A capability is labelled with exactly one of the following (a capability may carry a
second qualifier such as `production-blocked` where a production validator or code seal
also applies):

| Label | Meaning |
| --- | --- |
| **implemented-simulated** | Fully implemented and exercised end-to-end, but every effect is a simulated record in the control-plane database via the simulator plugin. No real infrastructure is created. |
| **implemented-fake-only** | Implemented and exercised, but only against in-process fakes / injected test doubles. The real adapter is not the shipped runtime path and no real endpoint is contacted. |
| **controlled-live-read-only** | A real, **read-only** integration wired into the shipped runtime that can contact an explicitly authorized target — but only after a complete, fail-closed gate chain passes. Sealed by default; never mutates. |
| **sealed-by-default** | The real implementation (or its architecture) exists in the tree but ships disabled and fail-closed: the shipped default composition refuses at the first privileged boundary. Reachable, if at all, only through an explicit, reviewed, deployment-local profile with every gate satisfied. |
| **contract-complete** | The durable schema, contracts, and code for a capability exist and are tested, but the capability is not wired into any shipped runtime path — it is dormant. |
| **partially-implemented** | Part of the capability exists; the remainder does not. |
| **not-implemented** | No implementing code exists. The capability appears, if at all, only as names/strings in schemas, seed data, or test fixtures. |
| **production-blocked** | Cannot be activated in production by design: a production-config validator (`Settings`) or a hard code seal refuses it. |

## D. Critical distinctions

These separations are real and enforced in code; conflating them misstates current truth.

- **Local topology canvas state is not a saved revision.** Client edits are not persisted until a save writes an immutable `TopologyRevision`.
- **A saved revision is not necessarily validated.** A saved revision is `draft`; validation is a separate, permissioned action.
- **Validation is not submission.** Submission is refused unless the revision is validated with a current matching result.
- **Submission is not approval.** Approve/reject is a separate, separately-permissioned, audited decision on a submitted revision.
- **Approval is a decision only.** Approving a topology revision records a decision and sets the approved-revision pointer — **nothing else**.
- **An approved topology revision is not yet a canonical published EnvironmentVersion.** Approving a topology revision does not publish a canonical EnvironmentVersion and does not generate a deployment plan.
- **EnvironmentVersion publication is not plan generation.** The plan generator binds to an `EnvironmentVersion`, not a topology revision.
- **Plan approval is not execution.** Approving a plan does not activate real provisioning (charter Invariant 5).
- **Controlled live read-only discovery is not provisioning, and is not the complete disposable-lab lifecycle.** It is read-only and imports no mutation code; the full real dry-run / apply / verify / destroy lifecycle does not exist yet.
- **Candidate plans produced by discovery are non-executable.** Live apply remains sealed.
- **The complete real dry-run / apply / verify / destroy lifecycle is not yet available.** It is sealed by default.
- **The B1-B disposable-lab architecture is locked (ADR-020) as a design-only architecture lock.** It defines the ordered real read-only qualification → real OpenTofu plan → exact change-set approval → exact prepared-plan apply → observed-state + isolation verification → separate destroy plan → separate destroy approval → destroy → zero-residue lifecycle, the operation-specific phased unsealing, the threat model, and the acceptance contract. **The architecture lock does not activate or authorize anything.** No real OpenTofu process has run, no real plan has run, no real apply or destroy has occurred, and no real Proxmox mutation has occurred. Both B1-A subprocess seals (`_B1A_SUBPROCESS_SEALED`) remain `True`, the controlled live read-only discovery status is unchanged, and all existing execution seals remain active.
- **B1B-PR2 implements real worker-local, filesystem-only toolchain attestation.** The `RealToolchainVerifier` performs bounded, containment-checked, race-detected on-disk verification of an explicit toolchain layout — executable identity, exact version, binary-integrity SHA-256, module-bundle tree hash, provider lockfile hash, offline provider-mirror tree hash, renderer version, the offline CLI configuration, the remote-state backend class, and runtime-download-disabled — against the immutable, secret-free profile. It **never executes the binary, reads PATH, opens a network socket, resolves a secret, touches the database or audit log, loads a provider plugin, contacts a remote-state service, or renders a workspace**, and it is **not wired into any execution path** — the runner and `run_real_provisioning` still default to the fake verifier. Tests use inert temporary fixtures only; **no real installed OpenTofu binary has been inspected as project evidence**, no OpenTofu process has run, no endpoint has been contacted, and all init/plan/apply/destroy capability remains sealed. A passing unit fixture is **not** evidence that an operator's future real toolchain is ready — actual deployment-local attestation remains a future operator action after reviewed installation. B1-B remains `partially-implemented`, `production-blocked`; both B1-A seals remain `True`; **B1B-PR3 (real read-only eligibility preflight) is now implemented (sealed by default); B1B-PR4 (remote-state + JIT secret readiness) remains the next slice**.
- **B1B-PR3 implements a controlled, worker-owned, read-only eligibility preflight — durably wired but default-disabled.** The worker seam `run_real_eligibility_preflight` composes the existing controlled read-only gate chain (approved+active onboarding, an approved+unexpired live-read authorization with matching connection/boundary hashes and current collector-contract/endpoint-allowlist versions, an approved worker identity) and — only through an explicitly-injected, separately-reviewed activation composition — reaches the read-only Proxmox transport (Path B, `run_live_readonly_collection`), evaluates a versioned, deterministic, provider-neutral eligibility policy over the bounded observations, and persists immutable, redacted, org/target/onboarding/authorization/worker-identity/policy-bound, hash-bound, expiry-bound `live_verified` evidence into the **existing** `TargetPreflight`/`TargetEvidenceRecord` tables (no parallel evidence table). **Durable execution path (in PR3, not deferred):** the API `request_eligibility_preflight` is **enqueue-only** — it durably queues a `WorkflowRun` + outbox row and **refuses inline execution with no in-process fallback**, never contacting a host, building a transport, resolving a secret, or persisting evidence; the worker registers a `EligibilityPreflightWorkflow` + `eligibility_preflight_activity` (Temporal, worker-only) that loads the authoritative records from a fresh worker session, checks cancellation / the deployment-local stop posture before any contact, and runs the seam. Live-evidence persistence is **structurally worker-originated**: the recorder lives in the worker package and the API cannot import it (the architecture-boundary lock forbids API→worker imports outside the dispatch seam and name-forbids the eligibility symbols), and it takes a TYPED evaluator result (never a caller-supplied evidence dict) — so the `(live_readonly_proxmox, live_verified)` label alone can never create live evidence. The shipped composition is fully **sealed** (no transport/resolver/collector), so the durable path runs to completion yet **refuses at the seal before any target contact**; the seal is the out-of-band reviewed composition, never an env flag. There is no automatic preflight on onboarding approval; a failed live path never falls back to simulated evidence. **The actual production collector, through the currently reviewed GET allowlist, yields `unverifiable`/`ineligible` — never `eligible`** (isolation posture / no-route, VM-ID collision, quota capacity, and storage disposability are never inferred; reaching `eligible` is a documented deployment prerequisite), proven by an integration test over the exact real chain; a fake-collector success in tests is **not** evidence of production-path eligibility. It runs **no OpenTofu**, mutates nothing, resolves no API-side secret, and constructs no real toolchain verifier or activation grant. **No real Proxmox host has been contacted, no OpenTofu process has run, and no real Proxmox mutation has occurred.** A passing test fixture is **not** evidence that a real target is eligible. B1-B remains `partially-implemented`, `production-blocked`; both B1-A subprocess seals (`_B1A_SUBPROCESS_SEALED`) remain `True`, and the architecture lock does not activate or authorize anything.
- **OIDC bearer-token verification is implemented (backend).** A presented bearer token is cryptographically verified (RS256 signature against the configured issuer's JWKS, exact issuer, required audience, time claims, bounded subject) and its exact `sub` maps to one pre-provisioned user; organization and permissions are database-owned and token roles/groups/email grant nothing.
- **Interactive browser login (Authorization Code + PKCE) is implemented (OIDC-B / ADR-018).** The public `secp-web` client (no secret) obtains an access token via the code+PKCE(S256) flow and sends it as `Authorization: Bearer`; `/api/v1/me` is the authoritative browser identity and token claims grant nothing. Tokens are session-scoped only (no localStorage/DB persistence, no `offline_access`). No browser **refresh token** is requested, retained, or used: the dev Keycloak client disables refresh-token issuance (`use.refresh.tokens=false`), and since omitting `offline_access` alone does not stop an ordinary refresh token, the frontend also strips any `refresh_token` before persisting the user and keeps only a refresh-token-free projection in memory (access-token expiry requires a fresh interactive login).
- **Production OIDC deployment guardrails, preflight & runbooks are implemented (OIDC-C / ADR-019).** A same-origin production model is locked (one canonical HTTPS public origin serves the web app and the `/api/` API); `SECP_PUBLIC_ORIGIN` is validated as an exact HTTPS origin; production CORS is disabled (empty allow-origins, stateless bearer, no cookies) while development allows only the exact configured origin without credentials/wildcards; a production Host allowlist (canonical + optional internal health host, never `*`) fails closed with no `/health` bypass; a token-free preflight (`python -m secp_api.oidc_preflight`) checks discovery/JWKS/issuer/keys without logging in, obtaining a token, or touching the DB/audit; the frontend API base is same-origin (never a localhost fallback in production); and placeholder-only reference guardrails (`infra/production/`) plus an operations runbook (`docs/runbooks/oidc-production.md`) are provided. This does **not** make the whole platform production-ready.
- **The development fallback identity is not production authentication.** It is a clearly-gated dev-only principal, evaluated only on a no-Authorization-header request, hard-refused in production, and must never be described as production auth. A presented bearer token is always verified first and never falls back.

## E. Current real-infrastructure boundary

There are two distinct real-infrastructure paths. They have different statuses and must not
be conflated.

### Path A — worker-owned SSH controlled read-only discovery (`controlled-live-read-only`)

This path is wired into the shipped worker discovery consumer, but it is **sealed by
default**. It can perform strictly read-only host contact against an explicitly authorized
Proxmox target **only** when the deployment-local, worker-owned controlled-integration
profile is enabled **and** the complete fail-closed gate chain passes, each gate enforced
before any host contact:

1. deployment-local controlled-integration profile enabled + valid admission material
   (endpoint, worker identity key/anchor, CA bundle);
2. exactly one approved worker identity for the org;
3. enrollment / onboarding drift check;
4. strict, descriptor-pinned mounted-bundle validation;
5. control-plane Ed25519 signed-nonce proof-of-possession admission (one-time);
6. endpoint-binding + live-read authorization re-verification;
7. host-key `known_hosts` pin proven before any SSH invocation;
8. post-probe one-time admission consume + worker-identity re-check.

With the profile disabled (the shipped default) discovery is sealed and contacts nothing.
The path imports no mutation-capable module and produces only a **non-executable** candidate
plan.

### Path B — HTTP target-evidence collector (`contract-complete`, dormant / sealed)

The hardened read-only HTTP transport, the closed read-only request policy, the collector,
the read-only-preflight orchestration, and the resolver-activation authorization contracts
are fully built and tested — but **not activated**. The HTTP target-evidence collector
remains dormant: its runtime gate is default-disabled and it has no shipped caller; the
read-only-preflight consumer ships a denying worker-identity verifier, a sealed activation
gate, and a sealed secret resolver, so it always terminates `credential_unavailable` and
contacts nothing. **The existence of transport and collector contracts must not be described
as activation.**

### Boundary statements (all currently true)

- **Real Proxmox provisioning remains unavailable.**
- **The OpenTofu subprocess remains hard-sealed** and is refused in production.
- Ownership observation and every other mutation gate remain fail-closed.
- **No UI or API approval alone activates infrastructure execution.**
