**Authoring convergence (ADR-016) — control-plane complete, simulated:** publishing an approved topology revision into a canonical immutable `EnvironmentVersion`, and planning against exactly that version, is now implemented end to end (PR A–E) as fail-closed, org-scoped control-plane logic on a **simulated** planning path — see [`ADR-016`](adr/ADR-016-authoring-convergence-environment-version-publication.md). Approval of a topology revision **publishes nothing automatically**; publication is one explicit, `version:publish`-gated, audited, idempotent mutation; a deployment plan binds **exactly one** immutable `EnvironmentVersion` (`environment_version_id` + `version_content_hash`) and surfaces its typed, server-owned publication provenance identically across generate → submit → approve, failing closed (redacted `plan_version_binding_invalid`) on any binding disagreement; legacy `v1alpha1` versions remain fully plannable with **null** provenance. Every step is a separate, explicit action — no publication auto-creates an exercise, no exercise auto-generates a plan, no transition auto-triggers the next, and approval deploys nothing. PR E added **no** worker/provider/transport/real-infrastructure code and **no** DeploymentPlan topology/provenance column or migration; real mutation remains sealed. The point-in-time PR-A…PR-D lines below are retained as historical progression.

**Backend OIDC bearer verification (ADR-017 / OIDC-A) — implemented:** the control-plane API now performs cryptographic OIDC access-token verification. A presented `Authorization: Bearer` token is accepted only when it is syntactically valid, uses the fixed `RS256` allowlist, its signature verifies against the configured issuer's JWKS, its `iss` exactly matches configuration, the configured API `aud` is present, it is unexpired with a valid `nbf`/`iat` (bounded clock skew), and its exact `sub` maps to exactly one **pre-provisioned** internal user (`app_user.subject`, now globally unique for non-null values via a partial index). Organization, roles, and permissions come **exclusively** from SECP database records — the token's roles, groups, email, or organization claims grant **nothing**, and there is **no** just-in-time user creation. A Bearer header is always evaluated before the development fallback, and an invalid token never falls back to dev-admin; the dev fallback remains available only on a no-header request in non-production. Authentication failures are closed and redacted (`401 {"error":{"code":"unauthenticated"}}` + `WWW-Authenticate: Bearer`; a distinct `503 {"error":{"code":"authentication_unavailable"}}` when discovery/JWKS is unavailable), never revealing the cause; logs carry only a bounded reason category, never the token/claims/subject. The interactive browser login that obtains those tokens is implemented by **OIDC-B** (below). This slice establishes the trusted backend boundary only; it unseals **no** provisioning or infrastructure mutation, and authentication remains distinct from the still-authoritative per-route authorization checks.

**Browser Authorization Code + PKCE login (ADR-018 / OIDC-B) — implemented:** the placeholder browser login is replaced by a real, provider-neutral OpenID Connect **Authorization Code flow with PKCE (S256)** through the public `secp-web` client (no client secret), using the maintained `oidc-client-ts` library. The browser reads a **public, secret-free** `GET /api/v1/auth/config` (server-derived `mode`, issuer, public client id, audience, fixed `openid profile email` scope with **no** `offline_access`, and fixed relative callback paths), runs the code+PKCE flow, and sends the resulting **access token** as `Authorization: Bearer` to the API — the ID token is never an API credential and never grants org/role/permission. `/api/v1/me` is the **authoritative** browser identity; token claims are never used for SECP authorization or organization selection. Tokens are **session-scoped only** (no localStorage/IndexedDB/cookies/DB persistence, no service-worker cache); access-token expiry clears local state and requires a fresh interactive login (no silent renewal, no `offline_access`, no refresh-token session). Every application route is guarded (no protected content or domain API call before authentication resolves); an invalid callback, `state`/`nonce` mismatch, token error, or API **401** fails closed to `/login` with a sanitized same-origin return path, a **403** keeps the session, and a **503** shows a provider-unavailable message. Logout clears local state and invokes the provider end-session when available. Frontend auth errors are bounded categories that never expose a token, code, state, nonce, verifier, claim, subject, or provider detail. The development fallback remains available **only** when the backend reports `mode = dev_fallback` (non-production). This slice adds **no** backend session/BFF, token database, JIT provisioning, token-derived RBAC, or multi-issuer support, and unseals **no** provisioning or infrastructure.

**Production OIDC deployment guardrails, preflight & runbooks (ADR-019 / OIDC-C) — implemented:** the authentication architecture is now safely deployable and operationally documented. A **same-origin** production model is locked (one canonical HTTPS public origin serves the web app and the `/api/` SECP API; browser callback/logout URLs derive from it): `SECP_PUBLIC_ORIGIN` is validated as an exact HTTPS origin, production **CORS is disabled** (empty allow-origins; the API uses stateless bearer tokens, never cookies) while development allows only the exact configured origin without credentials or wildcards, and a production **Host allowlist** (canonical host + optional internal health host, never `*`) fails closed with no `/health` bypass. A token-free operator **preflight** (`python -m secp_api.oidc_preflight`) reuses the OIDC-A hardened HTTP seam to check discovery/JWKS, exact issuer agreement, HTTPS endpoints, and a usable RSA signing key — without logging in, obtaining a token, or touching the database/audit log — with stable exit codes. The frontend API base is **same-origin** in a browser build (never a localhost fallback in production). Placeholder-only production reference guardrails (`infra/production/`) and a full operations runbook (`docs/runbooks/oidc-production.md`) are included. This **does not** certify the whole platform as production-ready: pre-provisioned internal identities remain required, real provider mutation and the complete real disposable-lab lifecycle remain sealed or incomplete, and approval never becomes execution because authentication is deployable. There is **no** first-class production identity-lifecycle API/UI/CLI in this slice — a direct DBA `subject` binding is outside the SECP application mutation path and is **not** SECP-audited (creates no `AuditEvent`); required `sub` bindings must exist and be independently verified before rollout, and a transactional, authorization-gated, SECP-audited identity-administration workflow remains future work.

**B8 production read-only discovery activation (SECP-PR5F) — implemented in the repository, not installed or exercised:** the narrow `secp_discovery_activation` package makes the existing worker-owned B8 read-only path safely deployable through closed `inspect`/`plan`/`render`/`install`/`verify`/`status`/`rollback`/`evidence` operations, fixed worker-private persistent state, a worker-only `/var/run/secp` mount, strict internal server-authenticated HTTPS admission, authenticated non-secret evidence, and receipt-bound ordinary-worker recreation/rollback. Its runtime is explicitly **not image-only**: it retains the exact old reviewed worker base image and overlays a complete content-addressed `secp_api` + `secp_worker` ZIP through a fixed read-only mount and exact `PYTHONPATH`; the controller instead requires a new digest-qualified API image and verified Alembic head `d8f1a2b3c4e5`. Fixed controller/worker base Compose files are content/metadata CAS-bound, actual API/proxy/worker runtime projections are reobserved, the private listener IP is separated from DNS SNI/SAN identity with one exact worker `extra_hosts` binding, and controller/worker installation is joined by a detached-signed fixed outbox/inbox handoff chain. It reuses `WorkerDiscoveryNode`, `ProxmoxReadOnlyBootstrapSession`, the existing browser Read-Only Bootstrap wizard/script, worker admission, endpoint binding, bundle manager/runtime, mounted-bundle validation, and the discovery engine; it creates no second enrollment system. The wizard now composes the existing worker-identity lifecycle into one exact-node, three-review approval/link transaction before the separate live-read authorization/bind gate.

The `d8f1a2b3c4e5` PostgreSQL migration installs and validates a durable named `CHECK` fence that rejects new `ed25519_signed_nonce` registrations throughout the signed controller/worker handoff. Runtime rollback requires the exact current, complete live role-local journal: after a preliminary compatibility read, the exact transaction-owned API generation or still-mounted worker overlay engages the fence, and internal compensation rebinds and re-engages it immediately before mutation; the Alembic downgrade independently canonicalizes and retains that fence while old code is live. The second explicit controller install durably commits and independently authenticates aggregate evidence while the fence remains engaged, then releases it and freshly proves both the complete evidence chain and the exact live released state. Evidence-committed/fence-engaged is a resumable `awaiting-finalization` state; a restart after release reauthenticates evidence and reobserves the fence without releasing twice. `verify` and `status` remain read-only and never engage or release it. Old key bindings use the prior-image-compatible `refused` status.

Public runtime evidence uses a value-redacted configuration digest; the complete Docker configuration is instead protected by a domain-separated HMAC kept only in the root-owned `0600` journal and compared in constant time. Two-sample mount isolation resolves no-follow device/inode/type identities, so textual aliases, bind mounts, hardlinks, symlinks, unresolved sources, protected-path ancestry, and identity drift refuse. Worker bundle assembly also requires the session descriptor's recorded SSH public-key fingerprint to match the current local worker key before touching the bundle. Live discovery scopes uniqueness to Ed25519 signed-nonce identities, while the composite same-label review refuses and preserves any active non-Ed25519 collision.

The read-only Proxmox strap already exists outside this change, but its installed key cannot be assumed to match a surviving worker private key: after fresh persistent worker-key generation, the existing idempotent wizard script is the **only** key-rotation/binding path. PR5F was not installed, did not recreate a worker, import production artifacts, run that script, SSH, contact Proxmox/Temporal/PostgreSQL/OpenBao/a state backend, or submit a workflow. The controlled-live operator is absent and its package installation is next; no controlled-live plan composition is installed, no real OpenTofu plan has run, apply/destroy remain unavailable, and PR6 remains frozen.

## A. Status taxonomy

A capability is labelled with exactly one of the following (a capability may carry a
second qualifier such as `production-blocked` where a production validator or code seal
also applies):

| Label | Meaning |
| --- | --- |
| **implemented-simulated** | Fully implemented and exercised end-to-end, but every effect is a simulated record in the control-plane database via the simulator plugin. No real infrastructure is created. |
| **implemented-fake-only** | Implemented and exercised, but only against in-process fakes / injected test doubles. The real adapter is not the shipped runtime path and no real endpoint is contacted. |
| **controlled-live-read-only** | A real, **read-only** integration wired into the shipped runtime that can contact an explicitly authorized target — but only after a complete, fail-closed gate chain passes. Sealed by default; never mutates. |
| **sealed-by-default** | The real implementation (or its architecture) exists in the tree but ships disabled and fail-closed: the shipped default composition refuses at the first privileged boundary. Reachable, if at all, only through an explicit, reviewed, deployment-local profile with every gate satisfied. |
| **contract-complete** | The durable schema, contracts, and code for a capability exist and are tested, but the capability is not wired into any shipped runtime path — it is dormant. |
| **partially-implemented** | Part of the capability exists; the remainder does not. |
| **not-implemented** | No implementing code exists. The capability appears, if at all, only as names/strings in schemas, seed data, or test fixtures. |
| **production-blocked** | Cannot be activated in production by design: a production-config validator (`Settings`) or a hard code seal refuses it. |

## D. Critical distinctions

These separations are real and enforced in code; conflating them misstates current truth.

- **Local topology canvas state is not a saved revision.** Client edits are not persisted until a save writes an immutable `TopologyRevision`.
- **A saved revision is not necessarily validated.** A saved revision is `draft`; validation is a separate, permissioned action.
- **Validation is not submission.** Submission is refused unless the revision is validated with a current matching result.
- **Submission is not approval.** Approve/reject is a separate, separately-permissioned, audited decision on a submitted revision.
- **Approval is a decision only.** Approving a topology revision records a decision and sets the approved-revision pointer — **nothing else**.
- **An approved topology revision is not yet a canonical published EnvironmentVersion.** Approving a topology revision does not publish a canonical EnvironmentVersion and does not generate a deployment plan.
- **EnvironmentVersion publication is not plan generation.** The plan generator binds to an `EnvironmentVersion`, not a topology revision.
- **Plan approval is not execution.** Approving a plan does not activate real provisioning (charter Invariant 5).
- **Controlled live read-only discovery is not provisioning, and is not the complete disposable-lab lifecycle.** It is read-only and imports no mutation code; the full real dry-run / apply / verify / destroy lifecycle does not exist yet.
- **Candidate plans produced by discovery are non-executable.** Live apply remains sealed.
- **The complete real dry-run / apply / verify / destroy lifecycle is not yet available.** It is sealed by default.
- **The B1-B disposable-lab architecture is locked (ADR-020) as a design-only architecture lock.** It defines the ordered real read-only qualification → real OpenTofu plan → exact change-set approval → exact prepared-plan apply → observed-state + isolation verification → separate destroy plan → separate destroy approval → destroy → zero-residue lifecycle, the operation-specific phased unsealing, the threat model, and the acceptance contract. **The architecture lock does not activate or authorize anything.** No real OpenTofu process has run, no real plan has run, no real apply or destroy has occurred, and no real Proxmox mutation has occurred. Both B1-A subprocess seals (`_B1A_SUBPROCESS_SEALED`) remain `True`, the controlled live read-only discovery status is unchanged, and all existing execution seals remain active.

## E. Current real-infrastructure boundary

There are two distinct real-infrastructure paths. They have different statuses and must not
be conflated.

### Path A — worker-owned SSH controlled read-only discovery (`controlled-live-read-only`)

This path is wired into the shipped worker discovery consumer, but it is **sealed by
default**. It can perform strictly read-only host contact against an explicitly authorized
Proxmox target **only** when the deployment-local, worker-owned controlled-integration
profile is enabled **and** the complete fail-closed gate chain passes, each gate enforced
before any host contact:

1. deployment-local controlled-integration profile enabled + valid admission material
   (endpoint, worker identity key/anchor, CA bundle);
2. exactly one approved worker identity for the org;
3. enrollment / onboarding drift check;
4. strict, descriptor-pinned mounted-bundle validation;
5. control-plane Ed25519 signed-nonce proof-of-possession admission (one-time);
6. endpoint-binding + live-read authorization re-verification;
7. host-key `known_hosts` pin proven before any SSH invocation;
8. post-probe one-time admission consume + worker-identity re-check.

With the profile disabled (the shipped default) discovery is sealed and contacts nothing.
The path imports no mutation-capable module and produces only a **non-executable** candidate
plan.

PR5F adds the repository-owned `secp_discovery_activation` deployment package around this existing
path. It fixes worker-private host persistence at `/var/lib/secp/discovery-worker`, mounted read-write
only into the ordinary worker at `/var/run/secp`; renders the exact existing B8 environment contract;
provides a dedicated internal HTTPS admission listener with a worker-pinned CA and exact route
allowlist; and transactionally recreates/restores only the ordinary worker. Admission still uses the
Ed25519 signed-nonce proof above — it is not client-certificate mTLS. Its revalidating lifecycle
states distinguish `disabled`, `prepared`, `TLS-ready`, `worker-recreation-required`,
`worker-starting`, `keys-generated`, `public-node-published`, `awaiting-finalization`, `awaiting-bootstrap-session`,
`awaiting-proof`, `awaiting-authorization`, `awaiting-bundle`, `bundle-ready`,
`discovery-contacted`, and `recovery-required`; enabled flags alone never mean ready.

The package has not been installed or exercised: the deployed ordinary worker's B8 flags remain
false, it has no `/var/run/secp` persistent mount/keys/bundle, and no B8 durable records exist. The
existing external read-only Proxmox strap was not created by PR5F and no host was contacted. Once an
installation generates its fresh persistent worker key, the old target key is not trusted: an
operator must select the published node in the
existing Read-Only Bootstrap wizard, create a fresh session, and run that wizard's idempotent script
as the sole key-rotation/binding mechanism. No second enrollment or target bootstrap contract exists.

### Path B — HTTP target-evidence collector (`contract-complete`, dormant / sealed)

The hardened read-only HTTP transport, the closed read-only request policy, the collector,
the read-only-preflight orchestration, and the resolver-activation authorization contracts

read-only-preflight consumer ships a denying worker-identity verifier, a sealed activation
gate, and a sealed secret resolver, so it always terminates `credential_unavailable` and
contacts nothing. **The existence of transport and collector contracts must not be described
as activation.**
